# Respond to the First Meridian Disruption

Stable content ID: `contract.meridian.respond-first-disruption.v1`
Content type: contract definition
Version: 1.0.0-draft
Maturity: draft
Family: crisis and operational response
Scope: player and country within a game session
Story arc: `story-arc.global.meridian-corridor.v1`
Stage: capacity warning or customs security intrusion
Implementation status: field mapping pending

## Player-facing summary

Recommend a response to the active disruption affecting the Meridian Corridor and your country.

## Supported disruption families

- port congestion;
- harvest and food-price pressure;
- mineral export delay;
- reservoir and energy warning;
- skilled-labor or component constraint;
- customs and payment-verification intrusion.

## Objective

Choose a response that addresses immediate harm while explaining cost, capacity, and delayed risk.

## Required output

1. Identify the confirmed problem.
2. Identify one important uncertainty.
3. Select a response.
4. Explain the immediate benefit.
5. Explain the immediate cost.
6. Identify one delayed consequence.
7. Define one measure used to judge whether the response worked.

## Learning objectives

- supply shocks;
- risk management;
- decision-making under uncertainty;
- leading and lagging indicators;
- short-term and long-term policy effects.

## Eligibility

- relevant event active;
- required briefing delivered;
- player authenticated;
- no approved completion for the same event instance.

The contract must reference the runtime event instance, not only the reusable event definition.

## Completion rubric

Approved when:

- confirmed fact and uncertainty are distinguished;
- response is feasible for the assigned country or institution;
- benefit and cost are both stated;
- delayed consequence follows logically;
- success measure is observable.

Changes requested when:

- response relies on unsupported authority;
- player assumes unconfirmed attribution;
- consequence is unrelated to the action;
- response ignores the assigned country’s dependency.

## Reward policy

Reward band: substantial crisis response.

Final amount: pending combined economy and urgency review.

Urgency should not automatically multiply reward unless the policy is visible and exploit-safe.

## Narrative effects

Approval may:

- mark crisis-response participation;
- influence character follow-up;
- unlock a reflection contract;
- contribute to achievement or reputation progress if supported.

It does not automatically resolve the event unless the runtime system explicitly aggregates approved actions.

## Failure and expiry

The contract expires with the event decision window.

An accepted submission arriving near expiry requires one consistent policy:

- allow submission through a grace deadline;
- freeze new acceptance but preserve accepted work;
- or cancel transparently without penalty.

## Admin guidance

Evaluate reasoning with the information available at the time. Do not penalize a player solely because a later event reveals a different outcome.

## Validation

- runtime event-instance reference valid and session-owned;
- player cannot select another session’s event;
- reward issued once;
- expiry and accepted-work policy defined;
- no tactical cybersecurity detail required;
- response options remain country appropriate.
